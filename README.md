# SQL-решение для построения хронологии событий и анализа жизненного цикла подписок на основе модели конечного автомата (State Machine)

## Описание проекта
Проект посвящен решению сложной продуктовой и инженерной задачи: восстановлению хронологии жизненного цикла пользователей подписочного сервиса (SaaS). 

В реальном бизнесе данные о подписках часто хранятся в виде пассивных временных интервалов (периодов действия). Для расчета ключевых продуктовых метрик (удержания Cohort Retention Rate, оттока Churn Rate, численного баланса активных пользователей по дням) эти интервалы необходимо преобразовать в динамический поток событий перехода пользователя из одного качественного состояния в другое.

В рамках проекта реализован оптимизированный SQL-запрос, который восстанавливает хронологическую ленту событий на основе теории конечных автоматов (State Machine).

## Исходные данные и ожидаемый результат

### Исходная таблица `subscriptions`
Содержит историю всех подписочных периодов для каждого клиента:
- `customer_id` — уникальный идентификатор клиента;
- `membership_start_date` — дата начала подписочного периода;
- `membership_end_date` — дата окончания подписочного периода;
- `membership_status` — статус подписки (`Free`, `Paid`, `Non-member`).

**Пример исходных данных:**
| customer_id | membership_start_date | membership_end_date | membership_status |
|-------------|-----------------------|---------------------|-------------------|
| 115         | 2020-01-01            | 2020-02-15          | Free              |
| 115         | 2020-02-15            | 2020-03-15          | Paid              |
| 115         | 2020-03-15            | 2020-04-01          | Non-member        |
| 115         | 2020-04-01            | 2020-10-01          | Paid              |

### Ожидаемый результат
Хронологически упорядоченный список событий переходов для каждого клиента:
- `customer_id` — уникальный идентификатор клиента;
- `change_date` — дата изменения состояния;
- `event` — тип продуктового события.

**Пример результата:**
| customer_id | change_date | event      |
|-------------|-------------|------------|
| 115         | 2020-01-01  | WarmStart  |
| 115         | 2020-02-15  | Convert    |
| 115         | 2020-03-15  | Cancel     |
| 115         | 2020-04-01  | ColdStart  |
| 115         | 2020-10-01  | Cancel     |

## Правила переходов (Матрица состояний)
Бизнес-логика классификации событий построена по следующим правилам переходов:
- `Free` → `Paid`: `Convert` (Конверсия в платящего)
- `Paid` → `Free`: `ReverseConvert` (Переход с платной на бесплатную)
- `Paid` → `Non-member`: `Cancel` (Cancel-событие промежуточного или полного завершения подписки)
- `Free` → `Non-member`: `Cancel` (Отказ от бесплатной подписки)
- `Non-member` → `Paid`: `ColdStart` (Активация платной подписки новым/вернувшимся пользователем)
- `Non-member` → `Free`: `WarmStart` (Активация бесплатной подписки после паузы)
- `Paid` → `Paid`: `Renewal` (Продление платного периода)
- `Free` → `Free`: `Renewal` (Продление бесплатного периода)

## Технические сложности и архитектура решения
Основная сложность задачи заключается в генерации терминального (конечного) события оттока на дату `membership_end_date` самой последней строки истории, избегая при этом создания дублирующих или ложных событий на внутренних стыках смежных периодов.

Алгоритм решения спроектирован с использованием передовых практик написания кода (производительность оценивается как O(N log N) благодаря отсутствию тяжелых джоинов):
1. **Изоляция финальной даты:** С помощью аналитической оконной функции `ROW_NUMBER()` определяется строго последняя запись окончания подписки для каждого пользователя.
2. **Формирование единой хронологической шкалы:** Даты начала всех периодов и единственная финальная дата прекращения подписки объединяются в один поток через `UNION ALL`.
3. **Расчет переходов в одном проходе:** С помощью оконной функции `LAG()` сопоставляются текущие и исторические состояния пользователей.
4. **Валидация бизнес-правил:** В конструкции `CASE WHEN` отсеиваются неизменившиеся состояния и формируется чистая витрина событий.

## Код решения (PostgreSQL)

```sql
WITH last_subscriptions AS (
    -- Шаг 1: Находим самую последнюю запись для каждого клиента.
    -- Это нужно, чтобы извлечь финальную дату окончания всей истории подписок (membership_end_date).
    -- Используем ROW_NUMBER(), чтобы отсортировать даты окончания по убыванию.
    SELECT 
        customer_id,
        membership_end_date,
        ROW_NUMBER() OVER (
            PARTITION BY customer_id 
            ORDER BY membership_end_date DESC
        ) AS rn
    FROM subscriptions
),

timeline_union AS (
    -- Шаг 2: Формируем единую временную шкалу.
    -- Объединяем все даты начала периодов из исходной таблицы и добавляем к ним 
    -- только одну финальную точку закрытия контракта.
    -- В этот момент статус пользователя меняется на 'Non-member'.
    SELECT 
        customer_id,
        membership_start_date AS change_date,
        membership_status
    FROM subscriptions
    
    UNION ALL
    
    SELECT 
        customer_id,
        membership_end_date AS change_date,
        'Non-member' AS membership_status
    FROM last_subscriptions
    WHERE rn = 1
),

transitions AS (
    -- Шаг 3: Сопоставляем текущий статус с предыдущим.
    -- Оконная функция LAG() позволяет "посмотреть" на статус из предыдущей строки хронологии.
    -- Для самого первого события, когда предыдущей строки нет, подставляем 'Non-member'.
    SELECT 
        customer_id,
        change_date,
        LAG(membership_status, 1, 'Non-member') OVER (
            PARTITION BY customer_id 
            ORDER BY change_date ASC
        ) AS pr_st,
        membership_status AS cur_st
    FROM timeline_union
),

classified_events AS (
    -- Шаг 4: Применяем бизнес-логику для классификации событий.
    -- Конструкция CASE WHEN сопоставляет пары переходов с целевыми названиями событий.
    SELECT 
        customer_id,
        change_date,
        CASE
            WHEN pr_st = 'Free' AND cur_st = 'Paid' THEN 'Convert'
            WHEN pr_st = 'Paid' AND cur_st = 'Free' THEN 'ReverseConvert'
            WHEN pr_st = 'Paid' AND cur_st = 'Non-member' THEN 'Cancel' 
            WHEN pr_st = 'Free' AND cur_st = 'Non-member' THEN 'Cancel'
            WHEN pr_st = 'Non-member' AND cur_st = 'Paid' THEN 'ColdStart'
            WHEN pr_st = 'Non-member' AND cur_st = 'Free' THEN 'WarmStart'
            WHEN pr_st = 'Paid' AND cur_st = 'Paid' THEN 'Renewal'
            WHEN pr_st = 'Free' AND cur_st = 'Free' THEN 'Renewal'
        END AS event
    FROM transitions
)

-- Шаг 5: Получаем итоговый результат.
-- Фильтруем строки, оставляя только значимые изменения статуса (где event IS NOT NULL).
-- Сортируем данные по требованиям ТЗ: сначала по клиенту, затем по дате изменения.
SELECT 
    customer_id,
    change_date,
    event
FROM classified_events
WHERE event IS NOT NULL
ORDER BY 
    customer_id ASC,
    change_date ASC;

-- Шаг 5: Получаем итоговый результат.
-- Фильтруем строки, оставляя только значимые изменения статуса (где event IS NOT NULL).
-- Сортируем данные по требованиям ТЗ: сначала по клиенту, затем по дате изменения.
SELECT 
    customer_id,
    change_date,
    event
FROM classified_events
WHERE event IS NOT NULL
ORDER BY 
    customer_id ASC,
    change_date ASC;
```

## Используемый стек технологий
* **СУБД:** PostgreSQL (решение совместимо с большинством современных реляционных СУБД)
* **Аналитические инструменты:**
  * CTE (Common Table Expressions) для разделения логики на понятные шаги и обеспечения высокой читаемости кода
  * Оконные функции (`LAG()`, `ROW_NUMBER()`) для работы со смещением строк и ранжированием дат
  * Объединение множеств (`UNION ALL`) для бесшовной стыковки временных шкал
  * Условные операторы (`CASE WHEN`) для классификации продуктовых состояний
